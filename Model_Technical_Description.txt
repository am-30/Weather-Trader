KBOS DAILY MAXIMUM TEMPERATURE MODEL — TECHNICAL DESCRIPTION
For Third-Party Quantitative Audit
As of: March 29, 2026
================================================================================

This document describes the complete quantitative pipeline that produces the
edge table probabilities for the Kalshi KXHIGHTBOS Daily Maximum Temperature
market (Boston Logan Airport, KBOS). It is intended to be sufficiently detailed
that an auditor could independently reconstruct the current probability output,
verify the calibration, and identify faults or biases.

The pipeline has five stages:
  1. Data ingestion (ASOS observations, NWP forecasts)
  2. Kalman filtering (real-time temperature and model-bias estimation)
  3. NWP blending (weighted combination of HRRR/GFS/ECMWF)
  4. Monte Carlo simulation (Ornstein-Uhlenbeck daily-maximum distribution)
  5. Edge calculation (MC probabilities vs. Kalshi market prices)

Calibration runs daily at 10:05 AM ET after the NWS CLI settlement is confirmed,
updating all OU and Kalman parameters from the rolling historical window.

SECTION 1 — MARKET DEFINITION AND SETTLEMENT
================================================================================

Market: Kalshi KXHIGHTBOS
Target variable: Official NWS daily maximum temperature at KBOS (Fahrenheit),
                 as reported in the NWS Boston Climate Summary (CLI).

Settlement source: NWS CLI KBOS, published at approximately 10 AM ET the
following calendar day. The CLI reports the official daily maximum rounded to
the nearest integer degree Fahrenheit. This is the authoritative value and may
differ from the ASOS tabular maximum by 0–1°F due to the ASOS 0.5°C persistence
filter (see Section 2.3).

Market structure: Kalshi lists a set of bucket markets per trading day, each
defined by a floor_strike and/or cap_strike in °F. The YES contract on bucket
B(floor, cap) pays $1 if floor ≤ official_max ≤ cap. All strikes observed to
date are decimal (e.g., 38.5, 46.5) stored as NUMERIC(5,1) in the database;
the cap and floor define integer intervals.

Example buckets for a day with 49°F midpoint:
  B_bottom (cap=44.5):  YES if official_max ≤ 44°F
  B44.5    (44, 45):    YES if official_max ∈ {44°F, 45°F} (i.e., 44 or 45)
  B46.5    (46, 47):    YES if official_max ∈ {46°F, 47°F}
  B48.5    (48, 49):    YES if official_max ∈ {48°F, 49°F}
  B50.5    (50, 51):    YES if official_max ∈ {50°F, 51°F}
  B_top    (floor=52):  YES if official_max ≥ 52°F

Two-phase settlement:
  Phase 1 (7:00 PM ET): Market status set to "settled" using max(ASOS readings)
    as preliminary final_official_high. Auto-trading disabled.
  Phase 2 (10:05 AM ET next day): NWS CLI fetch overwrites preliminary value
    with the authoritative integer high. This triggers run_full_calibration().

The cli_settlement_confirmed flag in the markets table is set only when Phase 2
completes successfully. ONLY dates with cli_settlement_confirmed=TRUE are used
for model calibration to ensure integer NWS values are used, not ASOS floats.

SECTION 2 — ASOS OBSERVATION DATA
================================================================================

2.1 Source Hierarchy

The system polls KBOS ASOS observations every 2 minutes via four sources
in priority order, with a 4-minute rate-limit guard between API calls:

  1. NWS time-series API (PRIMARY):
       GET https://api.weather.gov/stations/KBOS/observations
         ?start={since_utc}&limit={N}
       N is gap-proportional (max 500). Returns all observations since the last
       stored reading. Approximately 1–3 minutes faster than IEM on recent data.

  2. IEM Mesonet bulk (SECONDARY):
       Returns all readings since last stored timestamp in a single CSV request.
       Useful fallback when NWS API is unavailable.

  3. Aviation Weather Center METAR (TERTIARY):
       aviationweather.gov/api/data/metar for KBOS. Provides SPECI special
       observations on significant condition changes.

  4. NWS /observations/latest (LAST RESORT):
       Returns only the single most recent observation.

The 4-minute rate-limit guard ensures ≤15 real API calls/hour regardless of the
2-minute scheduler firing. A module-level _last_asos_fetch_utc timestamp enforces
this; if the interval has not elapsed, the most recent DB reading is returned
without touching any API.

2.2 ASOS Data Volume (as of March 29, 2026)

  Total readings stored: 4,162
  Coverage: January 22, 2026 through March 29, 2026 (~67 days)
  Station: KBOS (Boston Logan International Airport)
  Fields stored: observation_time_utc, temperature_f, dew_point_f,
                 wind_speed_mph, raw_metar

2.3 ASOS Persistence Filter — Critical Sensor Behavior

KBOS uses a 0.5°C persistence filter before reporting. The sensor only reports
a new temperature when the change since the last report exceeds 0.5°C. This
produces "quantized" Fahrenheit values in multiples of 0.9°F:

  Celsius → Fahrenheit (clean Celsius conversions observed in data):
    2.0°C = 35.6°F
    2.5°C = 36.5°F
    3.0°C = 37.4°F
    3.5°C = 38.3°F
    4.0°C = 39.2°F
    4.5°C = 40.1°F
    5.0°C = 41.0°F
    5.5°C = 41.9°F
    6.0°C = 42.8°F
    7.0°C = 44.6°F
    8.0°C = 46.4°F
    9.0°C = 48.2°F
    10.0°C = 50.0°F
    11.0°C = 51.8°F
    12.0°C = 53.6°F
    13.0°C = 55.4°F
    14.0°C = 57.2°F
    15.0°C = 59.0°F
    16.0°C = 60.8°F
    17.0°C = 62.6°F

Consequence 1: Whole-number Fahrenheit values (39°F, 40°F, 41°F, etc.) almost
never appear in the raw tabular feed because they do not correspond to clean
0.5°C Celsius values.

Consequence 2: The ASOS tabular max typically UNDERSTATES the true daily maximum
by 0.2–0.4°F on average. The NWS CLI reports the true peak (often captured in
the METAR 6-hour maximum field) rounded to the nearest integer. Over 8 confirmed
CLI dates, this gap (final_official_high − max_asos_reading) was:

  March 21: gap = 53 − 53 = 0°F
  March 22: gap = 47 − 46 = +1°F
  March 23: gap = 38 − 37 = +1°F
  March 24: gap = 43 − 42 = +1°F
  March 25: gap = 52 − 51 = +1°F
  March 26: gap = 63 − 63 = 0°F
  March 27: gap = 55 − 54 = +1°F
  March 28: gap = 38 − 37 = +1°F

Mean positive gap: +1.0°F (6 of 8 dates show +1°F gap)
NOTE: This is surprisingly uniform. The system currently applies a
persistence_filter_offset=0.30°F as a default. The empirical data above
suggests the effective gap might be better approximated at +0.75°F (mean of all
8 dates including zeros). This is a potential calibration bias — see Section 8.

Consequence 3: Kalman sigma calibration uses hourly-bucket diffs (not 5-minute
diffs) to average through the 0.9°F step artifacts. See Section 7.2.

2.4 Hard Floor Maintenance

current_max_observed in the markets table is the running ASOS maximum for the
trading day, updated atomically via:

  UPDATE markets
  SET current_max_observed = GREATEST(current_max_observed, :val)
  WHERE target_date = :date

This single-statement form is critical — no preceding SELECT. The hard floor
is passed to the MC as the minimum possible daily maximum: P(max ≥ hard_floor) = 1.

The 6 PM ET rollover resets the hard floor tracking to the next calendar day.
A hard floor catch-up query runs on app startup to backfill any missed readings
if the app was down during trading hours.

Example ASOS time series for March 26, 2026 (temperature °F by ET hour):
  00:00–02:54: 42.8°F (plateau — persistence filter, Celsius ≈ 6°C)
  02:54–04:04: 41.0°F (step down — Celsius crossed 5.5°C threshold)
  04:05–05:34: 42.8°F (step up — back to 6°C)
  05:35–06:59: 44.6°F (stepped up — 7°C)
  08:25–09:04: 46.4°F → 48.2°F → 50.0°F (morning ramp, 0.9°F steps)
  10:05–11:04: 51.8°F → 53.6°F → 55.4°F → 57.2°F (rapid pre-noon heating)
  12:00–16:35: 60.8°F → 62.6°F plateau (peak period)
  Final official NWS high: 63°F (captured in 6h max field, not tabular)
  ASOS tabular max: 62.6°F (gap = 63 − 62.6 = 0.4°F, rounded at NWS to +1°F integer)

This example illustrates why the NWS official high (63°F) exceeds the ASOS
maximum (62.6°F): the true peak of 63°F briefly exceeded the sensor's 0.9°F
threshold window and was captured in the 6-hour max field but appeared as only
62.6°F in the tabular feed.

SECTION 3 — NWP FORECAST DATA AND BLENDING
================================================================================

3.1 Data Sources

Three NWP models are fetched hourly from Open-Meteo via the free API:
  HRRR:  High-Resolution Rapid Refresh (NOAA) — 3 km US grid, updated hourly
  GFS:   Global Forecast System (NOAA) — pure GFS global, updated every 6h
  ECMWF: European Centre for Medium-Range Weather Forecasts — updated every 12h

API parameter: timezone="America/New_York"
CRITICAL: All hourly curves are returned in Eastern Time from midnight ET.
hourly_temps[0] = midnight ET, hourly_temps[10] = 10 AM ET, etc.
Using UTC hour as an index would introduce a systematic 4-hour shift (UTC−4 = EDT).

The predicted_daily_high per model is max(hourly_temps[:24]).
ECMWF curves have length 26 (includes 2 hours into next day); HRRR/GFS have 24.
The system caps indexing at nwp_len−1 to handle different curve lengths.

3.2 NWP Data Volume

  HRRR:  1,576 fetches across 67 days (January 22 – March 29, 2026)
  GFS:   1,576 fetches
  ECMWF: 1,577 fetches

3.3 Model Weights

Model weights are calibrated from Brier scores of the daily-high probability
predictions against confirmed CLI final_official_high values. The calibration
uses the first NWP fetch in the [10 AM, 1 PM) ET window per day (morning
forecast; see Section 7.4 for lookback bias discussion).

Current model weights (as of March 29, 2026, after morning calibration):
  HRRR:  0.3435
  GFS:   0.2916
  ECMWF: 0.3649

A guard prevents calibration when fewer than 14 qualifying dates exist
(dates with both final_official_high confirmed AND morning NWP available).
With fewer than 14 dates the weights are held at equal thirds (1/3 each).

3.4 Blended NWP Curve Construction

For each ET hour h (0–23), the blended temperature is:

  blended[h] = Σ(weight_m × curve_m[h]) / Σ(weight_m for models with data at h)

The normalization by Σ(weight_m) handles missing hours (e.g., HRRR ends at h=17
in some fetches). This per-hour normalization is currently NOT implemented — the
system truncates all curves to the shortest model's length. This is an open issue
(Section 8, item 8).

3.5 Morning Forecast Window

The "morning forecast" used for calibration purposes is defined as the first
NWP fetch with fetched_at_utc in the [10 AM, 1 PM) ET window. This is the
forecast made before the daily maximum typically occurs (~2 PM ET). Using any
later fetch would introduce lookback bias (models partially "see" the temperature
trend as it develops, especially intraday HRRR updates). See Section 7.4.

SECTION 4 — KALMAN FILTER
================================================================================

4.1 State Vector and Observation Model

The Kalman filter tracks two quantities jointly:
  x = [[dT_t], [B_t]]

  dT_t: Temperature DEPARTURE from the blended NWP forecast at the current
        ET hour (°F). This is NOT the absolute temperature — it is how far
        reality differs from the model.

  B_t:  Persistent NWP model bias (°F). Learned slowly from many updates;
        carried forward across trading days via warm-start.

Absolute temperature estimate:
  T_abs = nwp_current_hour + dT_t + B_t

Observation model:
  z = asos_temp − nwp_current_hour   (departure from NWP at observed moment)
  H = [[1.0, 1.0]]                   (both state components contribute to z)
  So:  z = dT + B  (+ observation noise)

This formulation ensures B_t is directly observable from every ASOS reading.
With the prior H = [[1.0, 0.0]] (used before March 2026), bias was unobservable:
  K[1] ≈ 0 (Kalman gain for bias row was near zero from the start)
  Result: bias converged extremely slowly, staying near 0.0 for weeks

With H = [[1.0, 1.0]] (current):
  K[1] ≈ 0.42 on the first observation tick
  Bias starts updating immediately, converging in hours rather than days

4.2 Filter Equations

Matrices:
  F = I₂ (2×2 identity)      — no state transition (departure is stable)
  H = [[1.0, 1.0]]            — observation matrix
  Q = [[Q_temp, 0], [0, Q_bias]]  = [[0.10, 0], [0, 0.05]]   (process noise)
  R = [[R_obs]] = [[0.40]]        (observation noise)

Predict step (triggered once per NWP hourly update):
  x_{k|k-1} = F × x_{k-1|k-1}  = x (no change — departure is stable)
  P_{k|k-1} = F × P × Fᵀ + Q  = P + Q (just inflates covariance)

  Note: When NWP changes between hours, the DEPARTURE dT is stable in
  expectation (both T_true and NWP shift by roughly the same amount). The
  predict step does NOT shift x — it only inflates P and updates the stored
  NWP reference used to compute T_abs.

Update step (triggered every 2 minutes on each new ASOS reading):
  z_k = asos_temp_k − nwp_current_hour
  y_k = z_k − H × x   (innovation: actual departure minus predicted departure)
  S   = H × P × Hᵀ + R  (innovation covariance — scalar for 1D observation)
  K   = P × Hᵀ × S⁻¹    (2×1 Kalman gain vector)
  x   = x + K × y        (state update)
  P   = (I − K×H) × P × (I − K×H)ᵀ + K × R × Kᵀ   ← Joseph form

Joseph form covariance update is used (rather than the simpler P = (I−KH)P)
because the standard form loses positive-definiteness after O(100) updates
due to floating-point accumulation. The Joseph form is numerically stable
across the ~288 updates per 24-hour day.

4.3 Noise Parameters (from settings.py)

  Q_temp = 0.10   (°F²/hour — departure process noise)
  Q_bias = 0.05   (°F²/hour — bias process noise; smaller → bias is "sticky")
  R_obs  = 0.40   (°F² — ASOS observation noise variance)

These are fixed hyperparameters, not calibrated from data. A Q_bias of 0.05
means the bias standard deviation grows by √0.05 ≈ 0.22°F per hour during the
predict step — moderately slow drift that allows the bias to change over hours
but not minutes.

4.4 Startup Behavior

Warm start (new trading day with yesterday's data available):
  dT₀ = 0.0
  B₀  = yesterday.kalman_bias_estimate
  P₀  = yesterday.kalman_covariance × 1.2   (20% inflation for daily uncertainty)

Cold start (first-ever run, no historical data):
  dT₀ = 0.0
  B₀  = 0.0
  P₀  = I₂ (identity)

Gap inflation (app restart after downtime > 0.5 hours):
  For each elapsed hour (up to 12), call predict(nwp_delta=0, dt=1.0) to inject
  accumulated process noise. This re-inflates P so K is large enough to correct
  large temperature errors quickly after restart. Without this, after many ASOS
  updates P collapses to ~0.01 and K ≈ 0.024, causing a 9°F innovation to move
  T_abs by only ~0.2°F per tick.

4.5 NWP Delta Clamping

On each hourly NWP update (predict step), the change in blended NWP from the
prior hour is clamped to ±5°F (kalman_max_nwp_delta setting). Changes larger
than 5°F/hr are treated as data corruption and clamped with a warning log.
This prevents a bad model run from destabilizing the filter.

SECTION 5 — MONTE CARLO SIMULATION
================================================================================

5.1 Process Model

The temperature T_t is modeled as an Ornstein-Uhlenbeck (OU) process with a
time-varying, NWP-anchored mean (attractor):

  dT = θ_t × (μ_t − T_t) × dt + σ_block × √dt × Z

  where:
    θ_t:    Mean-reversion speed (per hour) — time-varying by ET regime
    μ_t:    NWP attractor at time t (see 5.3)
    σ_block: Diffusion coefficient (°F/√hour) — time-varying by ET hour block
    dt:     5/60 hours (5-minute time step)
    Z:      Standard normal random variable

The simulation is discrete with dt = 5/60 h. The continuous-time OU stationary
variance is σ²/(2θ); this is used for the sigma cap (Section 5.4).

5.2 Simulation Parameters (Current — March 29, 2026)

Scalar theta (overnight/fallback): 0.2137/hr
  Half-life: ln(2)/0.2137 = 3.24 hours
  Interpretation: a temperature deviation from the NWP attractor decays to 50%
  of its initial value in ~3.2 hours.

Two-regime theta:
  theta_am (ET hours 6–13): 0.2896/hr  — half-life 2.39 hours
  theta_pm (ET hours 13–20): 0.2546/hr  — half-life 2.72 hours
  Overnight (0–5, 20–23): scalar theta = 0.2137/hr

  Physical interpretation: AM hours (solar heating) show faster reversion to the
  NWP target than overnight (advective, slow changes). PM is intermediate — near
  peak temperature the restoring force acts more like a thermostat.

Time-varying sigma (per ET-hour block):
  Block 0-6  (midnight–6 AM):  σ = 0.94°F/√hr
  Block 6-10 (6 AM–10 AM):    σ = 1.41°F/√hr   ← most volatile (morning ramp)
  Block 10-14 (10 AM–2 PM):   σ = 1.23°F/√hr
  Block 14-18 (2 PM–6 PM):    σ = 1.14°F/√hr
  Block 18-24 (6 PM–midnight): σ = 1.40°F/√hr   ← elevated (evening maritime)
  Scalar fallback (no block data): σ = 1.231°F/√hr

Persistence filter offset: 0.30°F
  Applied as an upward adjustment to the hard floor when initializing paths_max.
  Accounts for the expected ASOS-to-NWS gap described in Section 2.3.
  effective_floor = current_max_observed + 0.30

n_paths: 10,000 (default from settings)
n_steps: Computed as max(1, int(24.0 × day_fraction_remaining × 12))
  At ET hour 10 with 14 hours remaining:
    day_fraction_remaining = 14/24 = 0.583
    n_steps = int(24.0 × 0.583 × 12) = 168 steps

5.3 NWP Attractor Construction

At each simulation step s (at ET hour h = hour_offset + s/12):

  μ_t = nwp_curve[h] + nwp_anchor_offset + kalman_bias + drift_adj

NWP Anchor Offset:
  Purpose: T0 (current Kalman estimate) will generally differ from nwp_curve[h]
  at the start of the simulation. If this gap is projected forward as a constant
  offset, a low morning T0 will depress the entire afternoon forecast. The anchor
  weight ramps the offset from near-zero (early morning, far from NWP peak) to
  full (at/past the NWP peak, where the gap is informative).

  Algorithm:
    nwp_reference = nwp_curve[hour_offset]   (current NWP value)
    peak_hour_idx = argmax(nwp_curve)         (ET hour of blended NWP peak)
    hours_to_peak = max(0, peak_hour_idx − hour_offset)
    anchor_weight = 1 − hours_to_peak/peak_hour_idx  (if peak_hour_idx > 0)
                  = 1.0  (if peak_hour_idx = 0)
    raw_gap = T0 − nwp_reference
    gap_after_bias = raw_gap − kalman_bias   (subtract bias to avoid double-count)
    nwp_anchor_offset = gap_after_bias × anchor_weight

  Double-count fix: With H=[[1,1]], T0 = nwp + dT + B. The raw gap contains both
  dT (transient) and B (persistent bias). Since kalman_bias is also added to μ_t
  separately, subtracting it from the gap prevents B from entering μ_t twice.

  Worked example (March 29, 11:52 AM ET snapshot):
    hour_offset = 10 (10 AM ET)
    nwp_curve[10] = 37.45°F
    peak_hour_idx = 16 (ET hour of blended maximum 49.07°F)
    hours_to_peak = max(0, 16 − 10) = 6
    anchor_weight = 1 − 6/16 = 0.625
    T0 (Kalman) = 40.99°F
    kalman_bias = +0.25°F
    raw_gap = 40.99 − 37.45 = +3.54°F
    gap_after_bias = 3.54 − 0.25 = +3.29°F
    nwp_anchor_offset = 3.29 × 0.625 = +2.06°F

5.4 Sigma Cap

Purpose: Prevent the OU stationary std (σ/√(2θ)) from exceeding a physically
meaningful bound for daily temperature variability. Without the cap, a high
calibrated σ combined with the 5-minute dt produces a near-random-walk process
where per-step noise >> restoring force, inflating paths_max.

Formula:
  sigma_max = ou_max_stationary_std × √(2 × θ)
  sigma_used = min(sigma_calibrated, sigma_max)

5.5 Simulation Loop

Pseudocode for the main simulation loop:

  paths_current = [T0] × n_paths           (initialize all paths at T0)
  effective_floor = hard_floor + 0.30°F    (persistence filter adjustment)
  paths_max = [effective_floor] × n_paths  (initialize running maxima)

  pre-generate Z = n_steps × n_paths standard normal matrix

  pre-compute step_noise[s] for each step s (avoids branching in hot loop):
    et_hour_s = hour_offset + s × (5/60)
    block = et_hour_block(et_hour_s)
    sigma_block = sigma_by_block.get(block, scalar_sigma)
    step_noise[s] = min(sigma_block, sigma_max) × √dt

  pre-compute step_theta[s] for each step s:
    if 6 ≤ et_hour_s < 13: step_theta[s] = theta_am
    elif 13 ≤ et_hour_s < 20: step_theta[s] = theta_pm
    else: step_theta[s] = theta_scalar

  for step s in 0 .. n_steps-1:
    hour_idx = min(hour_offset + floor(s/12), nwp_len−1)
    μ = nwp_curve[hour_idx] + nwp_anchor_offset + bias + drift_adj
    dT = step_theta[s] × (μ − paths_current) × dt + step_noise[s] × Z[s]
    paths_current += dT
    if s >= bridge_steps:           (bridge_steps=0 for same-day simulations)
      paths_max = max(paths_max, paths_current)

  return paths_current, paths_max

5.6 Bridge Steps (Post-6 PM Rollover Only)

After the 6 PM ET rollover, the active trading date switches to tomorrow while
physical time is still tonight. The simulation uses a stitched NWP curve:
  stitched = tonight_curve[current_et_hour:24] + tomorrow_curve[0:24]
  bridge_hours = 24 − current_et_hour

During bridge_steps = bridge_hours × 12, paths evolve (so T0 reaches midnight
temperatures correctly) but paths_max is NOT updated. This prevents tonight's
temperatures (which are OUTSIDE tomorrow's NWS observation window) from locking
in paths_max above the plausible daily maximum for the new trading day.

Example: If T0 = 46°F at 9 PM on a warm evening and tomorrow's peak is forecast
at 38°F, without bridge protection paths_max would immediately be ≥46°F for all
paths, making P(max ≥ 44°F) = 100% even though tomorrow's actual high is forecast
at 38°F.

5.7 Output

run_simulation() returns (paths_current, paths_max) — both shape (n_paths,).

price_full_distribution() then computes for each strike K:
  P(paths_max ≥ K) = mean(paths_max >= K)

Distribution statistics also computed:
  percentile_10, _25, _50, _75, _90 of paths_max
  mean_max, std_max of paths_max

SECTION 6 — PROBABILITY PRICING AND EDGE TABLE CONSTRUCTION
================================================================================

6.1 CDF Representation

The simulation produces P(max ≥ K) for a discrete set of strikes K. Between
evaluated strikes, the CDF is linearly interpolated. Settlement boundaries
account for the NWS integer rounding: a continuous max of 39.6°F rounds to 40°F,
so the boundary between "≤39°F" and "≥40°F" is at 39.5°F.

The system simulates strikes at half-integer intervals (e.g., 39.5, 40.5, 41.5,
...) corresponding to the settlement boundary for each integer outcome.

6.2 P(YES) Calculation Per Market Bucket

Bottom bucket (cap=X, no floor):
  YES if official_max ≤ cap−1 (i.e., integer max rounds to < cap)
  P(YES) = 1 − P(max ≥ cap − 0.5)

Middle bucket (floor=X, cap=Y):
  YES if floor ≤ official_max ≤ cap
  P(YES) = P(max ≥ floor − 0.5) − P(max ≥ cap + 0.5)

Top bucket (floor=X, no cap):
  YES if official_max ≥ floor+1 (i.e., integer max rounds to > floor)
  P(YES) = P(max ≥ floor + 0.5)

  NOTE: The top bucket is Y if max > floor (strictly), because if max = floor
  exactly, the integer rounds to floor which is NOT ≥ floor+1. But since NWS
  reports integers, P(max = floor exactly in continuous space) ≈ 0, so this
  distinction is negligible in practice.

6.3 Normalization

After computing raw P(YES) for all buckets, the system checks if Σ P(YES) ≈ 1.0.
If |Σ − 1.0| ≤ 0.10, probabilities are normalized to sum exactly to 1.0.
If |Σ − 1.0| > 0.10, a loud warning is logged (structural problem — gaps in the
Kalshi market partition or a CDF that is not properly monotone).

6.4 Edge Calculation

For each active market (ticker) with Kalshi ask price A:
  edge = P(YES)_model − A

  Positive edge: model says YES is more likely than the market believes.
  Buy signal when edge exceeds a threshold (e.g., edge > 0.05).

  Negative edge: market overvalues YES.
  Sell signal when edge < −threshold (system currently only trades long YES).

Kelly fraction for position sizing:
  b = (1/ask) − 1              (net odds per $1 risked)
  kelly = (P × b − (1−P)) / b  (full Kelly fraction)
  contracts = min(0.25 × kelly × MAX_SIZE / (ask × 100), MAX_SIZE)

  The 0.25 fractional Kelly is a risk management choice (25% of full Kelly).
  MAX_SIZE is configurable; existing long positions are subtracted from Kelly
  contracts before placing new orders.

SECTION 7 — CALIBRATION

7.1 Schedule and Trigger

Calibration runs via run_full_calibration() at 10:05 AM ET each day, triggered
by job_confirm_settlement() after Phase 2 CLI settlement confirmation.
If the app restarts after 10:05 AM and yesterday's CLI was not confirmed,
a startup catch-up attempt is made.

run_full_calibration() calls in sequence:
  1. calibrate_sigma(target_date)
  2. calibrate_theta(target_date)
  3. calibrate_theta_by_regime(target_date)
  4. calibrate_intraday_drift(target_date)
  5. calibrate_model_weights(target_date)
  6. calibrate_brier_scores(target_date)
  7. calibrate_persistence_offset(target_date)
  8. calibrate_ou_max_stationary_std(target_date)  ← Phase 3, cold start now

All calibration functions share:
  lookback_days = 30 (rolling window)
  decay_tau = 10 days (exponential weighting)
  weight per date: w = exp(−days_back / 10)  where days_back = 0 (most recent) → 29

7.2 Sigma Calibration (calibrate_sigma)

Method: hourly-bucket diffs of ASOS temperature readings.

Step 1: Group ASOS readings into 1-hour buckets (nearest top-of-hour).
  A reading within 40 minutes of the top-of-hour is included; others excluded.
  This removes the 0.5°C persistence filter artifact: 5-minute diffs show 0.9°F
  jumps at sensor threshold crossings, inflating mean(dT²/dt) by 3–4×.
  Hourly bucketing averages through these jumps, recovering true hourly volatility.

Step 2: Compute consecutive hourly diffs, multiply by weighting:
  sigma² = Σ(w × dT²) / Σ(w × dt)
  where dt = 1 hour for hourly diffs

Step 3: Compute per-block sigmas using the same method, restricted to readings
  in each ET-hour block. Falls back to pooled sigma for blocks with <10 samples.

Step 4: Clamp sigma to [0.1, 1.5]°F/√hr.

Returns: (pooled_sigma, sigma_by_block_dict)

7.3 Theta Calibration (calibrate_theta and calibrate_theta_by_regime)

The mean-reversion speed θ is estimated via the discrete-time AR(1) representation
of the OU process:

  T_{t+1} − μ_{t+1} = (1 − θ×dt) × (T_t − μ_t) + ε_t
  ↔  Y = φ × X + ε,   where φ = 1 − θ×dt
  ↔  θ = (1 − φ) / dt

Practically, X_i = departure of T from blended NWP at time i
              Y_i = departure of T from blended NWP at time i+1

Weighted OLS through the origin:
  φ̂ = Σ(w × X_i × Y_i) / Σ(w × X_i²)
  θ̂ = (1 − φ̂) / dt

Exponential weights w_d = exp(−days_back/tau) applied per-day.

Two-regime calibration:
  Same method applied separately to AM pairs (hour ∈ [6,13)) and
  PM pairs (hour ∈ [13,20)). Falls back to (None, None) if either
  regime has fewer than 20 weighted-equivalent pairs.

7.4 Model Weight Calibration and Lookback Bias

Model weights are calibrated from Brier scores:
  BS_m = Σ(w_d × (p_m − outcome)²)

where p_m is the model's predicted probability of the YES outcome and outcome is
1 if the daily high fell in that bucket, 0 otherwise.

LOOKBACK BIAS FIX: p_m is computed from the FIRST NWP fetch in the [10 AM–1 PM)
ET window, not from the latest daily fetch. Earlier implementations used the most
recent NWP revision, which occurred after the daily maximum had already been
observed and effectively "looked ahead" (intraday HRRR revisions at 12 PM and
1 PM incorporate observed morning temperatures, partially predicting the afternoon).

The morning window (10 AM–1 PM) is the forecast made before the daily maximum
typically occurs (~2 PM ET), providing a genuine pre-event prediction.

Current model weights (8 qualifying dates, March 21–28):
  HRRR:  0.3435
  GFS:   0.2916
  ECMWF: 0.3649

  Note: ECMWF has the highest weight despite showing the largest individual-day
  errors (March 21: −4.0°F, March 24: +2.9°F, March 26: +0.6°F). This is because
  Brier scores reward calibration across the full distribution, not just the

SECTION 8 — KNOWN LIMITATIONS AND OPEN ISSUES
================================================================================

1. BLOCKING: No end-to-end validated live trading cycle.
   The system has never completed a full cycle with live Kalshi orders. All
   data paths (fetch → Kalman → MC → edge → order) are individually tested
   but the integrated live path is unvalidated.

2. BLOCKING: NWS CLI regex unverified against real HTML.
   fetch_official_daily_high() uses regex patterns against the NWS Climate
   Summary page. These patterns were written against expected format, not a
   confirmed live page. If the page structure differs, Phase 2 settlement
   will silently fail and final_official_high will remain at the ASOS
   preliminary value.

3. BLOCKING: _normalize_market() input validation absent.
   The Kalshi market normalizer assumes yes_bid_dollars/yes_ask_dollars are
   floats in [0,1]. If Kalshi changes the API to return integers in [0,100],
   all edge calculations would inflate 100× with no error raised.

4. Persistence filter offset clamp may be too tight.
   Empirical data (8 dates) shows an average ASOS-to-NWS gap of ~0.75°F
   (6 dates with +1°F gap, 2 with 0°F gap). The clamp of [0.0, 0.5]°F
   prevents the calibrated value from reaching what the data suggests.
   This may cause systematic underestimation of the hard floor, which
   reduces P(max ≥ hard_floor_zone) and biases the distribution downward.

5. Kalman bias on March 29 (+2.02°F) is suspect.
   The bias flipped from −2.15 to +2.02 between March 28 and March 29,
   coinciding with the bias explosion patch. A genuine reversal of this
   magnitude in 24 hours is unusual for Boston spring temperatures.

6. morning_drift (+1.15°F) and kalman_bias (+2.02°F) are additive and
   may be double-correcting the same underlying NWP bias.
   Both terms push μ_t upward. If the Kalman filter correctly absorbs the
   NWP bias over time (as intended), the drift should converge to zero.
   With a large bias AND a large drift, the attractor is shifted ~3.3°F
   above the blended NWP — verify this is warranted by observed temperature.

7. model_fair_value_prob = 1.0000 in all recent snapshots.
   Every snapshot shows model_fair_value_prob = 1.0 (100% probability of
   the YES outcome), which seems implausible. This may indicate that the
   snapshot is storing P(max ≥ hard_floor) which is always 1.0 by construction
   rather than P(max ≥ kalshi_strike). Needs investigation.

8. NWP blending truncates to shortest model curve.
   If HRRR provides 18 hours and GFS provides 24 hours, the blend is cut
   to 18 hours. The correct approach is per-hour blending using whatever
   models have data available at each hour.

9. First-startup hard floor gap.
   If the app starts mid-day with no prior ASOS readings in the DB, the
   hard floor starts at NULL (prior sentinel was −999, now NULL). The true
   daily maximum observed before startup is lost.

10. Test coverage gaps.
    Zero tests for: kalshi_fetcher.py, orchestrator.py, app.py,
    nws_cli_fetcher.py, db_manager.py, calibrate_ou_max_stationary_std().
    92 tests cover kalman, monte_carlo, phase1/2 calibration, ingestion.

11. Theta was highly unstable in early operation (March 15–26):
    March 15: θ = 2.00/hr (extreme — half-life 20 minutes, almost no persistence)
    March 17: θ = 0.62/hr
    March 20: θ = 0.19/hr
    March 26: θ = 0.41/hr
    March 27: θ = 0.21/hr (stabilized after more data)
    This instability is expected with <2 weeks of data. Parameters should not
    be trusted until at least 30 days of data have accumulated.
