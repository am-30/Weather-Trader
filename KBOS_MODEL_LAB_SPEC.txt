1. PRODUCT VISION AND USER STORIES
================================================================================

THE PROBLEM
-----------
The model has many tunable parameters (sigma, theta, drift, anchor offset,
persistence offset, sigma cap, cloud cover impact, etc.) and multiple
structural choices (drift on/off, anchor on/off, time-varying vs scalar
sigma, etc.). Currently there is no way to:

  a) See whether a parameter change improves or degrades accuracy.
  b) Compare two configurations side-by-side across all settled dates.
  c) Explore parameter sensitivity (what happens if I double theta?).
  d) Find the optimal value for a parameter given historical data.
  e) Do any of this without reading code or asking Claude Code.

THE SOLUTION
------------
A single Streamlit tab ("Model Lab") that functions as a quantitative
workbench. It has three modes:

  MODE 1 — REPLAY: Pick a scenario (preset or custom parameters), replay
  all settled dates, see aggregate accuracy metrics and per-date results.

  MODE 2 — COMPARE: Pick two scenarios (A and B), replay both across all
  settled dates, see side-by-side metrics with statistical significance.

  MODE 3 — OPTIMIZE: Pick a parameter to optimize, define a search range,
  let the system find the value that minimizes Brier score across history.

USER STORIES
------------
"I want to see whether removing drift from the attractor improves accuracy."
  → MODE 2: Scenario A = current production, Scenario B = preset "No Drift".
    Click "Run Comparison". See Brier score delta, calibration curves, per-date
    breakdown. See whether the improvement is statistically significant.

"I want to find the best sigma cap value."
  → MODE 3: Select "ou_max_stationary_std", range [0.5, 3.0], step 0.25.
    Click "Optimize". See a chart of Brier score vs parameter value. The
    minimum is highlighted. See a warning if the curve is flat (parameter
    doesn't matter much) or noisy (not enough data to determine optimum).

"I want to manually explore what happens with a very high theta."
  → MODE 1: Select "Custom" preset. Drag the theta slider to 0.8.
    Click "Run Replay". See how the model performs with fast mean-reversion.
    Compare mean_max predictions to actual highs on a scatter plot.

"I want to see if cloud cover adjustment helps on overcast days."
  → MODE 2: Scenario A = production (cloud cover off), Scenario B = preset
    "Cloud Cover Active" with adjustable inflation factor. See whether Brier
    improves specifically on high-cloudcover days.

"I want to see what parameters were active on a specific day and how the
 model performed hour-by-hour."
  → MODE 1: After running a replay, click on a date in the results table.
    See the hour-by-hour predicted distribution vs actual, the attractor
    curve with all corrections visualized, and the parameter values used.

Key principle: the replay engine reconstructs what the model WOULD HAVE
computed at a historical moment, then applies the scenario's parameter
overrides BEFORE running the MC. This means:

  - Historical ASOS data is used as-is (no override — this is ground truth).
  - Historical NWP curves are used as-is (no override — this is input data).
  - Historical Kalman state CAN be overridden (e.g., force bias=0 to test
    what happens without Kalman correction).
  - All MC parameters (sigma, theta, drift, anchor, etc.) CAN be overridden.
  - The actual NWS official high is used as the outcome for scoring.

3.1 Scenario Dataclass
-----------------------

@dataclass
class Scenario:
    """A complete set of parameter overrides for a backtest run."""
    name: str                            # Human-readable label

    # Structural toggles (on/off switches for model features)
    use_drift_in_attractor: bool = False  # Phase A default: off
    use_anchor_offset: bool = True
    use_time_varying_sigma: bool = True
    use_time_varying_theta: bool = True
    use_cloud_cover_adjustment: bool = False
    use_ensemble_spread_adjustment: bool = False
    use_persistence_offset: bool = True

    # Scalar parameter overrides (None = use historical calibrated value)
    sigma_override: float | None = None           # °F/√hr, replaces all blocks
    sigma_by_block_override: dict | None = None    # {block_label: sigma}
    theta_override: float | None = None            # /hr, replaces all regimes
    theta_am_override: float | None = None
    theta_pm_override: float | None = None
    ou_max_stationary_std_override: float | None = None
    persistence_filter_offset_override: float | None = None
    drift_am_override: float | None = None         # Only used if use_drift=True
    drift_pm_override: float | None = None
    kalman_bias_override: float | None = None       # Force a specific bias value
    anchor_weight_multiplier: float = 1.0           # Scale anchor_weight (0=off, 1=normal)

    # Cloud cover parameters (only used if use_cloud_cover_adjustment=True)
    cloud_cover_overcast_threshold: float = 80.0    # % above which = overcast
    cloud_cover_clear_threshold: float = 20.0       # % below which = clear
    cloud_cover_overcast_sigma_factor: float = 0.8
    cloud_cover_clear_sigma_factor: float = 1.1

    # Ensemble spread parameters
    ensemble_spread_threshold: float = 3.0          # °F
    ensemble_spread_sigma_factor: float = 1.3

    # MC parameters
    n_paths: int = 10_000
    random_seed: int = 42                           # Fixed seed for reproducibility

    # Run configuration
    eval_hours: list[int] = field(default_factory=lambda: [8, 10, 12, 14, 16])

The key design choice: every parameter defaults to None (use whatever was
historically calibrated), and the scenario only overrides what it explicitly
sets. This means "current production" is just Scenario(name="Production")
with all defaults — it replays using the historical calibrated values from
each date's system_state row.


3.2 ReplayResult Dataclass
---------------------------

@dataclass
class ReplayResult:
    target_date: date
    eval_hour: int

    # Inputs used (after scenario overrides)
    T0: float
    hard_floor: float
    effective_floor: float            # hard_floor + persistence offset
    sigma_used: dict[str, float]      # {block: sigma} actually used in MC
    theta_used: dict[str, float]      # {regime: theta} actually used
    bias_used: float
    drift_used: float                 # 0.0 if drift disabled
    anchor_offset_used: float
    nwp_predicted_high: float
    attractor_peak: float             # max(mu_t) across all steps

    # MC outputs
    mean_max: float
    std_max: float
    percentiles: dict[int, float]     # {10: val, 25: val, 50: val, 75: val, 90: val}
    strike_probs: dict[float, float]  # {strike: P(max >= strike)}
    market_probs: dict[str, float]    # {ticker: P(YES)} — normalized bucket probs

    # Scoring (filled in after MC, using known outcome)
    actual_high: float
    prediction_error: float           # mean_max - actual_high
    brier_components: dict[str, float]  # {ticker: (p_yes - outcome)^2}


3.3 ParameterizedReplayEngine
------------------------------

class ParameterizedReplayEngine:

    def replay_single(self, target_date: date, eval_hour: int,
                       scenario: Scenario) -> ReplayResult:
        """
        Replay one (date, hour) pair under the given scenario.

        Steps:
        1. Load historical data from DB:
           - ASOS readings up to eval_hour on target_date
           - NWP curves (latest fetch before eval_hour)
           - system_state for target_date (for calibrated params)
           - markets for target_date (for strikes and actual outcome)

        2. Reconstruct base MCParams via build_mc_params_historical():
           - T0 = latest ASOS reading before eval_hour (Kalman estimate
             if available from system_state, else raw ASOS)
           - hard_floor = max ASOS up to eval_hour
           - nwp_curve = blended curve from latest fetches
           - hour_offset = eval_hour
           - All other params from system_state

        3. Apply scenario overrides:
           - For each field in Scenario that is not None, replace the
             corresponding field in MCParams.
           - Handle structural toggles:
             * use_drift_in_attractor=False → set drift_adj=0.0 in MCParams
             * use_anchor_offset=False → set a flag so MC skips anchor calc
             * use_time_varying_sigma=False → collapse sigma_by_block to
               scalar (use the pooled sigma_volatility)
             * use_time_varying_theta=False → collapse theta_am/pm to
               scalar theta
             * anchor_weight_multiplier → multiply the computed anchor_weight
               by this factor (0.0 = no anchor, 0.5 = half, 1.0 = full)
           - Handle kalman_bias_override: if set, replace bias in MCParams
             with the override value (useful for testing "what if bias=0?")

        4. Run price_full_distribution(mc_params, strike_list).
           - strike_list from markets table for target_date.
           - If no markets stored, generate strikes from
             nwp_predicted_high ± 8°F in 0.5°F increments.

        5. Score against actual outcome:
           - actual_high from markets.final_official_high
           - For each market bucket: outcome = 1 if actual_high falls in
             bucket, 0 otherwise.
           - brier_component = (predicted_p_yes - outcome)^2

        6. Return ReplayResult with all fields populated.
        """

    def replay_scenario(self, scenario: Scenario,
                         start_date: date | None = None,
                         end_date: date | None = None,
                         max_days: int | None = None) -> list[ReplayResult]:
        """
        Replay all settled dates in the given range.

        Default: all dates where cli_settlement_confirmed=True.
        If max_days is set, use only the most recent N settled dates.

        For each date, replay at each hour in scenario.eval_hours.
        Use scenario.random_seed for reproducibility — same seed for
        all (date, hour) pairs within a scenario run.

        Returns list of ReplayResult sorted by (date, hour).
        """

    def compare_scenarios(self, scenario_a: Scenario,
                           scenario_b: Scenario,
                           **kwargs) -> ComparisonResult:
        """
        Run replay_scenario for both A and B, then compute paired metrics.
        Returns ComparisonResult (see below).
        """

@dataclass
class ComparisonResult:
    scenario_a_name: str
    scenario_b_name: str

    # Per-scenario aggregate metrics
    a_brier: float
    b_brier: float
    a_rmse: float
    b_rmse: float
    a_mean_bias: float               # mean(prediction_error)
    b_mean_bias: float
    a_calibration: list[tuple]       # [(bin_center, predicted_mean, observed_freq)]
    b_calibration: list[tuple]

    # Comparison metrics
    brier_diff: float                # a_brier - b_brier (negative = A is better)
    brier_diff_ci_low: float         # 95% CI lower bound
    brier_diff_ci_high: float
    brier_p_value: float             # from paired bootstrap
    rmse_diff: float
    is_significant: bool             # brier_p_value < 0.05

    # Per-date breakdown
    per_date: pd.DataFrame           # columns: date, a_brier, b_brier, a_rmse,
                                     # b_rmse, a_mean_max, b_mean_max, actual_high

    # Per-hour breakdown
    per_hour: pd.DataFrame           # columns: hour, a_brier, b_brier, ...

    # Raw results for drill-down
    a_results: list[ReplayResult]
    b_results: list[ReplayResult]


3.5 Metrics Module
-------------------
File: backtesting/metrics.py

def compute_aggregate_metrics(results: list[ReplayResult]) -> dict:
    """
    From a list of ReplayResults, compute:
      brier_score: mean of all brier_components across all (date, hour, ticker)
      rmse: sqrt(mean(prediction_error^2))
      mean_bias: mean(prediction_error) — positive = overpredict
      calibration_curve: list of (bin_center, predicted_mean, observed_freq)
        Bins: [0, 0.1), [0.1, 0.2), ..., [0.9, 1.0]
        For each bin: collect all (predicted_p, outcome) pairs where
        predicted_p falls in the bin. predicted_mean = mean(predicted_p),
        observed_freq = mean(outcome).
      sharpness: mean(|predicted_p - 0.5|)
      log_loss: -mean(outcome*log(p+eps) + (1-outcome)*log(1-p+eps))
      per_hour_brier: {hour: brier_score} for each eval_hour
      per_hour_rmse: {hour: rmse}
      per_hour_bias: {hour: mean_bias}
      n_dates: number of unique dates
      n_predictions: total (date, hour, ticker) triples
    """

def compute_paired_bootstrap(a_results, b_results,
                              n_bootstrap=1000, seed=42) -> dict:
    """
    Paired bootstrap test on Brier score difference.
    Resampling unit: DATE (not individual predictions).

    For each bootstrap iteration:
      1. Sample dates with replacement.
      2. Compute Brier score for A and B using only sampled dates.
      3. Record diff = A_brier - B_brier.

    Returns:
      mean_diff: mean of bootstrap diffs
      ci_low, ci_high: 2.5th and 97.5th percentiles
      p_value: fraction of samples where diff > 0 (if testing "is B better?")
    """

def compute_climatological_baseline(results: list[ReplayResult],
                                     historical_highs: list[float]) -> dict:
    """
    For each strike in the results, compute the historical frequency that
    the daily max exceeded that strike (from a multi-year climatology).
    Return same metrics dict as compute_aggregate_metrics.

    historical_highs: pulled from IEM or NOAA for the relevant month(s).
    If not available, skip this and note "climatology unavailable."
    """


3.6 Optimizer Module
---------------------
File: backtesting/optimizer.py

class ParameterOptimizer:

    def grid_search(self, base_scenario: Scenario,
                     param_name: str,
                     values: list[float],
                     metric: str = "brier") -> OptimizationResult:
        """
        For each value in values:
          1. Create a copy of base_scenario with param_name set to value.
          2. Run replay_scenario.
          3. Compute the target metric.

        Returns OptimizationResult with the full grid, best value, and
        overfitting warnings.

        Supported param_name values (must match Scenario field names):
          sigma_override, theta_override, theta_am_override, theta_pm_override,
          ou_max_stationary_std_override, persistence_filter_offset_override,
          drift_am_override, drift_pm_override, anchor_weight_multiplier,
          cloud_cover_overcast_sigma_factor, cloud_cover_clear_sigma_factor,
          ensemble_spread_threshold, ensemble_spread_sigma_factor

        Supported metric values: "brier", "rmse", "log_loss", "mean_bias_abs"
        """

    def multi_param_grid(self, base_scenario: Scenario,
                          param_grid: dict[str, list[float]],
                          metric: str = "brier") -> OptimizationResult:
        """
        Cartesian product grid search over 2-3 parameters simultaneously.
        WARNING: Exponential in number of parameters. Cap at 3 params and
        issue a warning if total combinations > 200.

        For 2 params with 10 values each = 100 runs × ~8 dates × 5 hours
        = 4,000 MC runs. At 10K paths each, this is ~40M path evaluations.
        Expect 3-8 minutes depending on hardware.
        """

@dataclass
class OptimizationResult:
    param_name: str                  # or comma-separated for multi-param
    param_values: list               # grid values tested
    metric_name: str
    metric_values: list[float]       # one per grid point
    best_value: Any                  # param value with best metric
    best_metric: float
    baseline_metric: float           # metric with the default/historical value

    # Overfitting diagnostics
    n_dates: int
    effective_degrees_of_freedom: int  # n_dates - 1
    is_overfitting_risk: bool         # True if n_dates < 3x len(param_values)
    overfitting_warning: str | None

    # For visualization
    grid_df: pd.DataFrame            # columns: param_value(s), metric, ±1SE

OVERFITTING SAFEGUARDS
-----------------------
With only 8-14 settled dates, any optimization is at high risk of overfitting.
The optimizer must communicate this clearly:

1. Always show the number of dates used and a warning threshold:
   - < 10 dates: "⚠️ HIGH overfitting risk. Results are indicative only.
     Do not deploy optimized parameters without 30+ dates of validation."
   - 10-20 dates: "⚠️ MODERATE overfitting risk. Interpret with caution."
   - 20+ dates: "Results are more reliable but still not definitive."

2. Show ±1 standard error bands on the metric curve. If the error bars
   overlap across most of the parameter range, the data cannot distinguish
   between those values — the "optimal" value is noise.

3. For multi-param grid search, warn when total grid points > n_dates * 3:
   "You're fitting {n_params} parameters from {n_dates} data points.
    This is almost certainly overfitting."

4. Compute leave-one-out cross-validation (LOO-CV) score alongside the
   in-sample score. For each date, optimize on all OTHER dates, then score
   on the held-out date. If LOO-CV is much worse than in-sample, overfitting
   is confirmed.

   LOO-CV implementation:
     For each held-out date d:
       For each param_value in grid:
         Run replay on all dates EXCEPT d with this param_value.
         Compute in-sample metric.
       Find best_value from in-sample.
       Run replay on date d with best_value.
       Record held-out metric.
     LOO-CV score = mean(held-out metrics across all dates).

   This is expensive (n_dates × n_grid_values × (n_dates-1) MC runs) but
   only needs to run when the user clicks "Validate" on an optimization
   result. Do not run automatically.

  Section 1 — Summary Metrics (st.metric cards in a row):
    Brier Score | RMSE | Mean Bias | Sharpness | N Dates

  Section 2 — Calibration Curve (Plotly scatter):
    X = predicted probability (bin center)
    Y = observed frequency
    Perfect calibration = diagonal line (plotted as reference)
    Points sized by number of predictions in each bin
    This is the SINGLE MOST IMPORTANT CHART for model quality.

  Section 3 — Per-Date Scatter (Plotly scatter):
    X = actual high (°F)
    Y = predicted mean_max (°F)
    Perfect prediction = diagonal line
    Points colored by date (or by eval_hour)
    Hover shows: date, hour, predicted, actual, error, sigma, theta used

  Section 4 — Per-Date Table (st.dataframe, sortable):
    Columns: Date | Hour | Predicted High | Actual High | Error |
             Brier | Sigma Used | Theta Used | Bias Used | Drift Used |
             Anchor Offset | Attractor Peak

    - The full predicted distribution (histogram of paths_max)
    - The attractor curve (mu_t over simulation hours) with the actual
      temperature overlay (ASOS readings)
    - The CDF with strike boundaries marked
    - All parameter values used for that specific (date, hour)

  Section 5 — Per-Hour Brier (Plotly bar chart):
    One bar per eval_hour showing that hour's Brier score.
    Shows whether the model is better in the morning vs afternoon.

  Section 6 — Error Distribution (Plotly histogram):
    Distribution of prediction_error (mean_max - actual_high) across all
    (date, hour) pairs. Should be centered near zero. Skew indicates
    systematic bias direction.

7. DATA FLOW AND CACHING
================================================================================

7.1 Caching Strategy
---------------------
MC simulations are expensive (10K paths × 168 steps × n_dates × n_hours).
Use Streamlit's @st.cache_data decorator to cache replay results.

Cache key: hash of (scenario, start_date, end_date, eval_hours).
The Scenario dataclass must be hashable — implement __hash__ based on
all fields. Since Scenario has mutable fields (lists, dicts), convert them
to tuples/frozensets for hashing.

Cache invalidation: automatic when any scenario parameter changes (different
hash). Also add a "Clear Cache" button in the UI for manual invalidation.

7.2 Performance Estimates
--------------------------
Per (date, hour) replay: ~0.5-1.0 seconds (10K paths, 168 steps)
8 dates × 5 hours = 40 replays = 20-40 seconds per scenario run
Comparison = 2 scenarios = 40-80 seconds
Optimization with 15 grid points = 15 scenarios = 5-10 minutes

These are acceptable per the user's preference ("accuracy matters more").
When the dataset grows to 30+ dates, consider:
  - Reducing default n_paths to 5,000 for optimization (still adequate
    for Brier score comparison — MC noise < model error)
  - Parallelizing date replays (Python multiprocessing or concurrent.futures)
  - Caching the historical data queries (ASOS, NWP, system_state per date)
    separately from the MC results, since the data queries are repeated
    across grid points in optimization.

7.3 Historical Data Prefetch
------------------------------
Before any replay run, prefetch ALL needed data in bulk:

  class ReplayDataCache:
      """Pre-loads all historical data needed for a date range."""
      def __init__(self, start_date, end_date, eval_hours):
          self.asos = {}      # {date: list[AsosReading]}
          self.nwp = {}       # {date: {model: NwpCurve}}
          self.states = {}    # {date: SystemState}
          self.markets = {}   # {date: list[Market]}
          self._load_all(start_date, end_date)

      def get_asos_up_to(self, target_date, hour) -> list:
          """Return readings for target_date with time <= hour."""
          return [r for r in self.asos[target_date]
                  if r.observation_time_utc.hour <= hour_to_utc(hour)]

This avoids N×M database queries during a grid search (where N=grid points,
M=dates×hours). The data is loaded ONCE and reused across all grid points.

APPENDIX: WHAT THIS REPLACES AND HOW IT CONNECTS
================================================================================

This feature REPLACES the ad hoc backtest tab that Claude Code previously
built. The existing tab should be removed (or renamed to "Legacy Backtest")
once Model Lab is functional.

This feature IMPLEMENTS Phase 0 from the original KBOS_SYSTEM_REVIEW.txt
(backtesting infrastructure) and Phase 4 (edge validation framework) in a
unified, user-facing tool.

This feature DEPENDS ON:
  - build_mc_params_historical() being correct and handling all edge cases
    (missing NWP, missing system_state, etc.)
  - price_full_distribution() accepting an MCParams object and returning
    the strike probability dict and paths_max statistics.
  - The markets table having final_official_high and cli_settlement_confirmed
    populated for settled dates.

This feature DOES NOT modify any production model code. It is a read-only
analysis tool that queries historical data and runs MC simulations with
parameter overrides. The only "write" action is caching results in
Streamlit's session state.

If the user finds an optimal parameter value via the optimizer, they should
manually update the production config (settings.py or system_state) — the
Model Lab does not auto-deploy parameter changes. This is intentional: the
human should make the final decision about whether to trust the optimization
given the overfitting risks.
